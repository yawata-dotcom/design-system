# 2台のPCで安全に並行作業する方法（作業がかぶらない・壊れない）

複数のPC（社長の2台運用や、社長＋スタッフ）で同時に開発しても、**作業がぶつからず・壊れない**ための共通ルールです。

> 言葉の言い換え：
> - **ブランチ** ＝ 作業用の「小部屋」。本流(main)を汚さずに自分の作業をためる場所。
> - **PR** ＝ 「この小部屋の変更を本流に入れてください」というお願い。番人(CI)が合格して初めて入る。
> - **pull** ＝ 最新を取り込む。**push** ＝ 自分の変更を送る。

---

## まず大前提（仕組みで守られていること）
全リポは **直接 main に書けません**（ブランチ→PR→番人(CI)緑→マージ のみ）。
だから「片方の作業がもう片方を上書きして消す」事故は起きません。残る注意は **同じファイルの同時編集**だけです。

### 自動の番人：重なりチェック（check-overlap）
**他の開いているPRと"同じファイル"を触るPRは、CIが赤になってマージできません。**
- ＝同じ画面（ファイル）を2つのPRが同時に本流へ入るのを実質的に禁止（人でなく"ファイルの重なり"で判定＝同じアカウント1人で2台でも効く）。
- 赤が出たら：先に相手のPRをマージ → このブランチで `git pull --rebase origin main` → 出し直す（重なりが消えれば緑）。
- ＝「壊れない」だけでなく「同じファイルの取り合いに気づかずマージ」も防げます。下のルールと二重で安全。

---

## 黄金ルール（どの分け方でも共通・これだけ守ればOK）
1. **作業前に必ず `git pull`**（本流を最新に）→ そのうえで**タスクごとに専用ブランチ**を切る。
2. **1ブランチ＝1タスク。同じブランチを2台で共有しない**（PCごと・作業ごとに別のブランチ）。
3. 反映は **ブランチ → PR → 番人(CI)緑 → マージ**。**小さく・こまめに**出す（大きく溜めない）。
4. マージされたら、もう一方のPCは **`git pull` で追いつく**。
5. 同じプロダクトを2台で触るときは、**担当する「画面」を先に分ける**（下の B 参照）。

---

## 分け方は2通り（状況で使い分け）

### A. PCごとに別プロダクト（いちばん安全）
例：1台目＝交通誘導警備、2台目＝建設。
- **別のリポ＝別の置き場なので、ファイルがぶつかりません。** 黄金ルールだけで完全に安全。

### B. 同じプロダクトを「画面ごと」に2台で分担
画面（ページ）は**1画面＝1ファイル**に分かれているので、担当画面を分ければ画面本体はぶつかりません。
- React製品（交通誘導警備・建設）：各画面は `app/src/app/<画面>/page.tsx`。
- **唯一ぶつかりやすいのは「サイドメニューの項目リスト」**（1ファイルに集約）。
  - React＝ `app/src/components/AppShell.tsx` の `MENU`

#### おすすめのやり方＝「事前シード方式」（メニューの取り合いを無くす）
1. **最初に1回だけ**、これから作る画面を**全部まとめて「近日公開（グレー）」でメニューに登録**する（PR1本）。
   - ＝項目だけ先に置く（まだ押せない灰色の状態）。`MENU` への大きな追記はこの1回で終わり。
2. 以降、各PCは **自分の担当画面のファイル(`page.tsx`)を作り**、その項目を**「押せる」に1行だけ変える**小さなPRを出す。
   - ＝メニューリストへの同時の書き込みがほぼ無くなる＝ぶつからない。
3. これで「2台が同じメニューファイルを奪い合う」状況が消えます。

#### 静的HTML製品（人材事業）は要注意
人材事業は画面の中身が**1ファイル（render.js など）に集まっている**ため、2台で同じ製品を分担すると
ぶつかりやすいです。やむを得ず分けるときは「画面（renderXxx）ごとに別ブランチ・同時編集を避ける・こまめにPR」。
**同じ製品の2台分担は、画面がファイル分割されている React 製品（交通誘導警備・建設）向き**です。

---

## もしぶつかったら（コンフリクト）あわてず直す
同じ場所を2人が変えると、マージ時に「コンフリクト」が出ます。多くは数行で簡単です。
```sh
git pull --rebase origin main   # 本流の最新を自分の上に重ね直す
# 表示された箇所（<<<<<<< ～ >>>>>>>）を見て、正しい形に直す
git add <直したファイル>
git rebase --continue
git push                        # 自分のブランチへ送り直す → PRが更新される
```
迷ったら手を止めて社長／担当に相談（無理に上書きしない）。
