# 標準エージェント（サブエージェント）定義

ここに置く `*.md` が「全社標準のサブエージェント」です。
`setup.sh` でこのフォルダの内容を各自の `~/.claude/agents/` に取り込みます。
＝全員が同じ専門エージェント（同じモデル割当）を使える状態になります。

## 今ある標準エージェント（役割×モデル割当＝2026-07-03 社長確定）

| 名前 | 役割 | モデル | いつ使う |
|---|---|---|---|
| `implementer` | 実装役（コードを書く） | Sonnet | 機能の実装 |
| `bug-hunter` | バグ取り役（ロジック・境界値・エラー処理） | Sonnet | 機能完成ごとのチェック班 |
| `security-reviewer` | セキュリティ役（権限すり抜け・漏えい・機微情報） | Opus | 同上＋認証/権限/公開系の変更時は必須 |
| `performance-reviewer` | 性能役（0.5秒死守・N+1・索引） | Sonnet | 機能完成ごとのチェック班 |
| `design-consistency` | 設計整合役（設計書・用語定義集との一致） | Sonnet | 機能完成ごとのチェック班 |
| `freee-integration` | freee連携役（会社ID固定・レート制限・名寄せ） | Sonnet | freee APIに触れる変更時 |
| `verifier` | 裏取り役（指摘を現物で再確認・誤検出落とし） | Opus | 並列レビューの後・毎回 |
| `review-chief` | 総括役（統合・見落とし探し・優先順位＝最後の関所） | **Fable** | レビューサイクルの最後に1回 |
| `design-guard` | 全社共通ルールの番人レビュー（check-*.sh実行＋報告） | Haiku | コミット・PRの前 |

### モデル割当の考え方（正＝esecurity-system `design/エージェントチーム_v2.md` v2.4）
- **決める場所（司令塔・総括役）＝Fable 5**：判断ミスが一番高くつく場所。総括役は1サイクル1回なので費用影響小。
- **事故を止める場所（セキュリティ・裏取り・Google番・設計役・法令番）＝Opus 4.8**：
  セキュリティ役をFable 5にしないのは、サイバー系作業の誤検知で止まるリスクがあるため。
- **実働（実装・通常チェック・テスト）＝Sonnet 5**：コーディング性能はほぼOpus級。
- **探す・定型（探索・マニュアル・見張り・design-guard）＝Haiku 4.5**：最安で十分。
- 定義ファイルの無い役割（探索役=Explore・設計役=Plan・マニュアル係など）は、司令塔が呼ぶ時に
  この表に沿ってモデルを都度指定する。
- 見直し条件：1〜2週間の使用量計測後／Sonnet 5の導入価格終了（2026-08-31）時。

## 追加・変更のルール
- 標準エージェントを足す/変える・**モデル割当を変える**のは **社長の承認（PR）**でのみ。
- 各自が勝手にローカルで別エージェントを増やしても、最終的な成果物は各製品のCI（番人）で
  チェックされるため、ルール違反はマージできません。標準をそろえる目的は「入口の統一」です。

## 形式（参考）
サブエージェント定義は、先頭にYAML（`name` / `description` / 任意で `tools` / `model`）を書き、
その下に役割の説明（システムプロンプト）を書いた Markdown です。`design-guard.md` を雛形にしてください。
`model` は `haiku` / `sonnet` / `opus` / `fable`（常にその系列の最新を指す）。
